# Project Lifecycle

This document defines how projects enter, grow inside, and leave Heptapus Open Code Organization.

## 1. Proposal

Project proposals are opened through the organization `.github` repository.

A proposal must explain:

- purpose;
- expected users;
- current maturity;
- license;
- maintainers;
- contribution model;
- security or safety considerations;
- existing links or prior art.

## 2. Review

Maintainers review:

- mission fit;
- maintainability;
- legal/license clarity;
- contributor safety;
- security risk;
- overlap with existing projects.

Outcomes:

- accepted;
- declined;
- needs revision;
- needs private follow-up.

## 3. Repository Setup

Accepted projects should receive:

- repository README;
- license;
- contribution guide;
- issue templates;
- security policy;
- code of conduct link;
- labels;
- roadmap or first milestone;
- maintainer ownership note.

## 4. Active Development

Active projects should keep:

- issues scoped and verifiable;
- PRs reviewable;
- docs current;
- maturity claims honest;
- security reports handled responsibly.

## 5. Maintenance Mode

A project can enter maintenance mode when:

- it is stable but low-change;
- maintainer capacity is limited;
- large roadmap work is paused.

The README should say what maintenance mode means.

## 6. Archive

Archive when:

- no maintainers remain;
- the project is superseded;
- legal/license uncertainty cannot be resolved;
- risk is too high for available maintainer capacity.

Archived repositories should keep a final status note.
