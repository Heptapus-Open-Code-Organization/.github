# Incident Response

This document covers security, safety, and conduct incidents at the organization level.

## Incident Types

- security vulnerability;
- leaked secret or credential;
- malicious code or dependency compromise;
- harassment or Code of Conduct violation;
- license/attribution issue;
- false maturity, benchmark, or safety claim;
- project ownership dispute.

## First Response

1. Acknowledge the report.
2. Preserve evidence.
3. Limit harm quickly.
4. Move private or sensitive details out of public issues.
5. Assign maintainers responsible for response.
6. Decide whether affected repositories need temporary restrictions.

## Security Vulnerabilities

Follow `SECURITY.md`.
Use `docs/SECURITY_RESPONSE.md` for detailed severity, secret-leak, fix, and disclosure steps.

For public issues containing exploit details or secrets:

- remove or redact sensitive content when possible;
- rotate leaked credentials immediately;
- open a private maintainer thread;
- publish a summary after a fix when appropriate.

## Conduct Incidents

Follow `CODE_OF_CONDUCT.md`.

Possible actions:

- warning;
- comment deletion;
- temporary interaction limits;
- repository permission removal;
- organization removal;
- GitHub report if platform abuse is involved.

## Post-Incident Notes

After resolution, document:

- what happened at a high level;
- impact;
- fix or response;
- follow-up tasks;
- whether policy/docs need updates.

Avoid publishing private reporter information.
