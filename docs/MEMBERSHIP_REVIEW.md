# Membership Review

Membership is not only access; it is trust and responsibility.

## Application Path

Applicants use the membership application issue form in `.github`.

Maintainers review:

- GitHub profile;
- desired role;
- background;
- contribution areas;
- relevant links;
- agreement with organization rules.

## Acceptance Criteria

An applicant can be accepted when:

- they agree to the Code of Conduct and contribution expectations;
- they have a clear intended contribution area;
- they show good-faith alignment with open-code work;
- maintainer capacity exists to onboard them;
- no obvious security, conduct, or license risk is present.

Experience is helpful but not required for every contributor role.

## Onboarding

Accepted applicants should receive:

- a welcome comment;
- a suggested first repository or issue;
- access level appropriate to their role;
- links to the Code of Conduct, Contributing Guide, and relevant project docs.

## Decline

Declines should be respectful and brief.

Possible reasons:

- unclear intent;
- no current maintainer capacity;
- mismatch with organization mission;
- unresolved security/conduct concern;
- proposal would be better as an external contribution first.

## Access Levels

Use least privilege:

- public contribution through issues/PRs requires no org membership;
- repository write access requires project maintainer trust;
- organization maintainer access requires operational responsibility;
- owner/admin access should be rare.
